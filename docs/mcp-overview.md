## TL;DR(一句话)

**让 AI 助手(Claude / Cursor 等)能像实习生一样,在你的思维导图上接手做扩展、整理、生成初稿——你审阅后接受或撤销, weekly 省 2-4 小时。**

---

## Part 1 · 做什么

### 一句话定义

把思维导图 app 升级为**人机协作工具**:你跟 AI 不能同时编辑同一份文档,但可以**轮流接管**——AI 干活时你看,你干活时 AI 等。

### 类比

**带一个实习生**:
- 你不会跟实习生同时改一份 Word,而是"你给任务 → 实习生做 → 你看 → 接受或打回"
- AI 就是那个实习生——能干活,但你最终拍板
- 实习生做错了一键撤销(不留垃圾)

### 关键特点

| 特点 | 含义 |
|------|------|
| 🔒 **互斥** | 同一时刻只有一个写者(人 or AI),不会互相覆盖 |
| ⚡ **实时可见** | AI 的每次操作你在画布上立刻看到 |
| ✋ **可中断** | 任何时刻你能接管,AI 不会锁死应用 |
| ↩️ **整段撤销** | AI 的一次会话作为 1 个 undo 单元,Cmd+Z 一次撤回 |

## Part 2 · 产生什么价值

### 5 个核心场景

| 场景 | 现状耗时 | 有 MCP 后 | 节省 |
|------|---------|----------|------|
| **会议纪要 → 思维导图** | 30-60 min 手动重建 | Claude 直接生成结构,你审阅微调 | **80%** |
| **主题扩展(打破 blank page)** | 卡 10-30 min 想子主题 | Claude 给 5-10 个候选,你挑 | **60%** |
| **重组乱糟糟的节点** | 1-2 小时体力活 | Claude 自动归类,你看结果 | **70%** |
| **Markdown 大纲 → mindmap** | 复制粘贴 30 min | Claude 直接读 md 生成 | **90%** |
| **mindmap → 汇报 markdown** | 手写 30 min | Claude 导出 | **80%** |

### 量化估算(假设用户每周做 3-5 个思维导图)

- **个人效率**:每周节省 **2-4 小时**
- **质量提升**:LLM 提供发散视角(用户想不到的子主题)
- **降低使用门槛**:不会用思维导图的人也能用(LLM 帮建结构)

## Part 3 · 为什么是这个架构

**核心命题**:架构不是技术堆砌,是产品诉求的工程表达。合适的架构 = **用最少的元件同时满足所有产品诉求**。

### 3.1 协作模式层(3 个权衡)

#### 权衡 1:为什么 App 内嵌,不是独立后台服务?

| 方案 | 体验 | 工程复杂度 |
|------|------|----------|
| **A. App 内嵌**(选) | AI 操作**实时可见**,协作感强 | 中 |
| B. Standalone 独立服务 | AI 写完你要 reload 才看到,延迟感强 | 低 |
| C. Web 多人实时协作 | 多人 + AI 同时改 | **极高**(OT/CRDT,5-10 倍工程量) |

**选 A 的理由**:"人机协作"的本质是**实时可见 + 随时接管**。B 是技术简单但产品体验差。C 是工程过重,ROI 不合算。

#### 权衡 2:为什么"互斥",不是"实时协作"?

- 互斥 = AI 操作时你看,你操作时 AI 等
- 实时 = 你跟 AI 同时改(需要冲突解决算法)

**理由**:同一份文档两个写者永远会冲突(参考 Git merge conflict)。你跟实习生也不会同时改一份文档。**互斥符合人机协作的真实场景,工程量节省 80%**。

#### 权衡 3:为什么"会话级",不是"操作级"?

- 会话级 = AI 进入"会话"期间 UI 锁定,会话结束才解锁
- 操作级 = 每次 AI 调用前后你都能动

**理由**:AI 扩展一个主题通常要 5-10 次连续调用。如果操作级,你跟 AI 互相插入,AI 看到的状态会前后不一致,undo 历史也会混杂。**会话级让"AI 时间"和"我的时间"清晰分开**。

### 3.2 架构层(5 个产品诉求 → 4 个架构元件)

#### 第 1 步:产品诉求推出架构约束

| 产品诉求 | 架构约束 |
|---------|---------|
| 🔒 数据不能丢 | 写入必须走**原子写 + 单一数据源**,不能有多条写入路径 |
| ⚡ LLM 操作实时可见 | LLM 操作必须**穿过 app**,不能绕过直接改文件 |
| ✋ 用户能随时接管 | 必须有**用户优先级最高**的中断机制 |
| 🚫 人机不互相覆盖 | 必须有**全局写者锁** |
| 🔐 AI 不能越权 | 必须有**前置权限检查** |

#### 第 2 步:5 个约束 → 必然推出当前架构

```
┌──────────────────────────────────────────────────────────┐
│  外部 LLM 客户端(Claude Desktop / Cursor)              │
└─────────────────────────┬────────────────────────────────┘
                          │
                     ① 本机 loopback
                     (🔐 AI 不能越权 — 只允许本机)
                          │
                          ▼
┌──────────────────────────────────────────────────────────┐
│  MCP Server(Tauri 内嵌 HTTP,localhost:23456)           │
│  ┌────────────────────────────────────────────────────┐  │
│  │  ② EditorMode Mutex(全局写者锁)                   │  │
│  │     • 人编辑时:LLM 调写 tool → 立即拒绝           │  │
│  │     • LLM 持锁时:人编辑 → UI 阻止 + 一键接管      │  │
│  │     (🚫 不互相覆盖 + ✋ 可中断)                   │  │
│  └────────────────────────────────────────────────────┘  │
└─────────────────────────┬────────────────────────────────┘
                          │
                     ③ Tauri event
                     (⚡ LLM 操作穿过 app)
                          │
                          ▼
┌──────────────────────────────────────────────────────────┐
│  App 主进程                                              │
│  ┌────────────────────────────────────────────────────┐  │
│  │  ④ 前端 store → mind-elixir → 画布                 │  │
│  │     • LLM 操作走跟人编辑**完全一样的路径**         │  │
│  │     • 自动复用现有的原子写、undo/redo、自动保存    │  │
│  │     (🔒 单一数据源 + 原子写)                      │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

**关键洞察**:**①②③④ 四个箭头**对应 4 个产品诉求的兜底元件,**每个元件只做一件事**。

#### 第 3 步:为什么这是**最合适**的架构?

**"合适"= 在所有可行方案中,用最少元件满足最多约束**。对比:

##### ❌ 架构 B:LLM 直接读写 .mmap 文件(Standalone)

```
LLM → MCP → 直接读写文件 ←─冲突─→ App 也在读写文件
```

- ❌ 两条写入路径,数据可能不一致
- ❌ LLM 写完用户要手动 reload,实时性丢失
- ❌ 需要文件锁,复杂且不可靠
- ✅ 唯一优点:app 不用开着
- **额外工程量**:+2-3 周

##### ❌ 架构 C:LLM 通过 HTTP 直接操作 store

```
LLM → HTTP → store API ──→ store 改了,但画布不知道
```

- ❌ 绕过 Tauri event,store 跟 mind-elixir 状态不一致
- ❌ 没有锁机制
- **额外工程量**:+1-2 周

##### ❌ 架构 D:Web 多人协作(OT/CRDT)

- ❌ 工程量是 A 的 **5-10 倍**
- ❌ 用户不需要"同时改"的场景
- **额外工程量**:+2-3 个月

##### ✅ 架构 A(选这个):LLM → MCP → Mutex → 标准 Tauri event → store

- ✅ **5 个约束全满足**(每个箭头兜底一个)
- ✅ **复用现有架构**:LLM 操作走的路径跟人编辑完全一样
- ✅ **最小元件数**:4 个核心元件
- ✅ **可证伪**:每个约束都有对应的失败信号

#### 这个架构最核心的设计哲学

> **"单一数据源"**:LLM 操作**不创建新的数据路径**,而是"伪装成一个特殊用户"走现有管道。
>
> 这意味着:**所有为"人编辑"做的安全保障(原子写、备份、undo、自动保存),LLM 自动享受**。

**一句话**:不是新造一套,而是把 LLM 接入既有的安全保障体系。

#### 代码层验证(设计已通过代码审查)

这个承诺不是宣传话术,是**代码层已验证**的事实:

| 承诺 | 代码层依据 |
|------|----------|
| LLM 走 mind-elixir 标准 API | `mind-elixir/dist/types/methods.d.ts` 确认 `addChild / reshapeNode / removeNodes / moveNodeIn` 都是 public API |
| 每个 API fire 标准 "operation" 事件 | `mind-elixir/dist/MindElixir.iife.js` 22 个 fire 点确认 |
| 现有 sync 链路自动复用 | `MindMapCanvas.tsx:323-328` 现有订阅 `operation` 事件 + `syncFromMindElixir` |
| 自动保存天然防抖 | `useAutoSave` 2s 防抖,LLM 60s 调 20 次 = 1 次保存 |
| undo 整合可行 | zundo `temporal.pause() / resume()` 已在现有 store 配置 |

**单一数据源验证脚本**(可在浏览器 DevTools 执行):
```javascript

// 预期:画布出现新节点 + dirty=true + 2s 后自动保存
```

## Part 4 · 如何证明这个架构合适

**核心理念**:"合适"不是测试通过,是 **"用户真的在用 + 真的省时间"**。

### Phase 1 · 只读 MVP(3-5 天)

**验证的架构元件**:① 本机 loopback + 标准 HTTP MCP 协议 → 🔐 安全诉求

- ✅ **成功标准**:Claude Desktop 能连上,问"我现在打开的思维导图有哪些节点"能正确回答
- ⚠️ **失败信号**:协议实现卡壳 > 2 天 → 评估技术方案
- 🛑 **中止条件**:Phase 1 都做不出来 = MCP 集成有重大障碍,整个方案中止

### Phase 2 · 互斥写(1-2 周)

**验证的架构元件**:② EditorMode Mutex + ④ 单一数据源 → 🔒 数据不丢 + 🚫 不冲突 + ✋ 可中断

- ✅ **成功标准**:
  - 用户给 Claude 会议纪要,60s 内生成 10-20 节点的思维导图
  - 用户能一键接管
  - Cmd+Z 一次撤销整个 LLM 会话
- ⚠️ **失败信号**:
  - 锁机制不稳(死锁/漏锁) → 重新设计互斥
  - 用户测试发现"AI 操作的节点 50%+ 需要撤销" → **AI 能力不足,不是架构问题**(可能换更大模型/改 prompt)
- 🛑 **中止条件**:用户不愿意把 AI 接入工作流 → 产品价值假设失败,重新思考场景

### Phase 3 · 协作体验(1 周 + 长期观察)

**验证的架构元件**:③ Tauri event 实时性 + UI 反馈 → ⚡ 实时可见

- ✅ **成功标准**:
  - 用户每周使用 **≥ 2 次**(留存)
  - 主动推荐给 **≥ 1 个同事**(NPS)
  - 单次协作节省 **≥ 30 分钟**(ROI)
- ⚠️ **失败信号**:用 1-2 次就不用 = 价值不够强,可能场景错了
- 🎯 **最合适的最终证明**:找 3-5 个真实用户用 2 周,看 usage log + 用户访谈(是否愿意推荐)

## 设计修正记录(2026-07-23 v1.1)

### undo 整合的真实行为

之前承诺"一次 Cmd+Z 撤销整个 LLM 会话"。代码层验证后,zundo 没有原生 wrap API,实际行为是:
- LLM 会话开始(acquired)→ `temporal.pause()`(操作不进 undo 历史)
- LLM 操作期间 → 画布正常刷新,但不污染 undo 历史
- LLM 会话结束(released/expired/forced)→ `temporal.resume()`
- 用户按 Cmd+Z → 撤回到**会话开始前的状态**(中间状态全部跳过)

**净效果**:Cmd+Z 一次跳过整个会话,但中间状态无法逐步撤销。如果用户想看 LLM 中间过程,需要操作历史侧栏(显示每一步,但只能整体撤销)。

**承诺差异**:从"撤销整段"细化为"跳过整段",语义更精确,但用户体验核心价值不变(用户能"取消"整个 LLM 会话)。
