# LLM 工程指引（精华版）

> 给 LLM 的工程开发思维框架。不是历史记录，是决策清单。
> 看到任何任务先读这份，再动手。

---

## 一、动手前必问的 7 个问题

每个任务开始前，逐条回答（不答清楚不动手）：

1. **用户的真实意图是什么？** 字面诉求 ≠ 真实意图。例：用户说"按钮不响应"，真实可能是"前面 5 步累积的状态导致按钮 disabled"。
2. **这个改动的爆炸半径多大？** 改 1 行 vs 重写模块；可逆 vs 不可逆；本地 vs 影响用户数据。
3. **失败时用户能回滚吗？** 没 git commit 就改 = 用户无法回退。
4. **测试覆盖了真实链路还是 mock 重？** 单元测试全过 ≠ 系统能跑。
5. **有没有"我假设了什么"？** 假设环境干净 / 假设依赖已装 / 假设字段已存在——任何假设都必须验证。
6. **这个 bug 的根因在第几层？** 表象（崩溃）→ 直接（字段 undefined）→ 工程（serde 配置）→ 系统（契约测试缺失）。修第 N 层，不只修第 1 层。
7. **修复后同类 bug 还会再发吗？** 如果会，加守则 / 测试 / 工具，不只修当前 case。

## 二、关键决策规则（if-then）

### 修 bug 时

| 如果... | 那么... |
|---------|---------|
| 用户报告"X 不工作" | **先复现**，再读代码。不要凭描述改代码。 |
| 看到 `undefined` / `null` 错误 | 不只是加 `?? []`。**追到数据源**：序列化？网络？类型声明？ |
| 修了一个字段 | **追问**：同类字段有没有同样问题？契约测试覆盖了吗？ |
| 测试通过但用户报 bug | **测试环境 ≠ 真实环境**。补 E2E 用真实事件序列。 |
| 看到 console error / network 500 | **立即追查**，不能"能跑就略过"。 |
| 第三方库行为异常 | **读它的源码**（dist/*.js），别假设文档对。 |
| `dispatchEvent` 模拟全过 | **不可信**。浏览器默认行为（contenteditable Enter=换行）模拟不出来。 |

### 写新功能时

| 如果... | 那么... |
|---------|---------|
| 加 `import "pkg/xxx"` | **立即验证**：看网络请求是否 200，看 console 是否报 specifier 错。 |
| 加新的数据字段 | **前后端类型同步 + 契约测试**（序列化输出必须含字段，即使空）。 |
| 加新的 user action | **必须有可见反馈**（视觉变化 / 状态指示）。无反馈 = 用户以为没响应。 |
| 加自动行为（auto-save / auto-edit） | **必须有可见退出方式**。隐性模式困用户。 |
| 加默认行为 | 问：**用户没主动触发时，这个行为合理吗？** 默认无副作用最安全。 |
| 加第三方依赖 | 看 `package.json` 的 `exports` 字段——可能限制了你能 import 的路径。 |

### 启动 / 部署时

| 如果... | 那么... |
|---------|---------|
| 启动命令失败 | **不要假设环境干净**。加 precheck（清理僵尸 / 验证依赖）。 |
| 端口 / 文件锁冲突 | **自动恢复**（kill + 重试），不只报错。 |
| 启动成功 ≠ 功能可用 | **必须模拟用户操作**，不能只看"进程起来了"。 |

## 三、已知陷阱信号（看到这些立即警惕）

| 信号 | 可能根因 |
|------|----------|
| `Cannot read properties of undefined (reading 'X')` | 数据契约破坏（字段在序列化/传输中丢失）|
| 用户报告"看起来正常但无法操作" | 焦点被困（input-box / modal 未关闭）|
| 单元测试全过但生产崩 | mock 重 + 缺契约测试 + 测试数据 ≠ 真实数据 |
| `dispatchEvent` 测试通过但用户报 bug | 浏览器默认行为差异（contenteditable / form / drag）|
| "Missing specifier in package" | package.json exports 限制，import 路径不在声明内 |
| 视觉模型说"图标大小不一" | 可能是浮动 UI 重复创建 + CSS layout 异常 |
| 修了同一类 bug 第 3 次 | 守则缺失，**写新守则**防止第 4 次 |
| `setState({...}, true)` 后 actions 消失 | replace 模式覆盖了整个 state（zustand）|
| 集成测试污染真实数据 | env var 重定向（如 `MINDMAP_TEST_DATA_DIR`）没生效 |
| `npm run xxx` 子进程找不到 cargo | PATH 没继承，脚本内显式 `export PATH=...` |

## 四、根因分析方法

**遇到 bug 永远按这个顺序，不要跳步**：

1. **复现**——用真实环境 / 真实事件序列。chrome-devtools MCP 是最接近真实的。
2. **看日志 / console**——开发模式 JSONL 日志含完整操作链路 + 错误堆栈。
3. **追到数据源**——错误位置 ≠ 根因位置。`undefined.children` 可能根因在 Rust serde 配置。
4. **读源码**——第三方库 bug 必须读 `node_modules/pkg/dist/*.js`，别假设文档。
5. **抽象归因**——这个 bug 属于哪类？契约 / UX / 环境假设 / 边界条件？
6. **加守则**——同类 bug 怎么防止再发？

**不要做的**：
- 看到 error 直接加 try-catch 吞掉
- 看到 undefined 直接加 `?? []` 不追源
- 看到"再试一次就好了"而不查为什么之前失败
- 复制 StackOverflow 答案不验证

## 五、验证清单（宣布"完成"前必跑）

```
[ ] 单元测试全过（前端 + Rust）
[ ] 集成测试全过（含契约测试）
[ ] TypeScript 编译通过
[ ] E2E 真实事件序列验证（不只 dispatchEvent）
[ ] 焦点残留检查（document.activeElement.isContentEditable）
[ ] console 无 error / warning
[ ] 网络请求全 200
[ ] 视觉模型评估（截图给 zai 分析）
[ ] 干净环境验证（不依赖残留状态）
[ ] 同类字段 / 同类场景已排查
```

**任何一项 ✗ 就不能宣布完成**。

## 六、UX 反 Pattern 识别

看到以下设计，**立即质疑**：

| Pattern | 问题 | 改进 |
|---------|------|------|
| 自动进入编辑/模态/聚焦 | 用户没主动触发 | 让用户主动 dblclick / F2 / 按钮 |
| 按钮无视觉反馈 | 用户不知道点了没 | 状态指示 + 选中高亮 |
| 唯一退出方式是快捷键 | 用户不知道键 | 加可见的 ✕ / 取消按钮 |
| 失焦自动保存无提示 | 用户不知道保存了 | 状态栏显示"已保存 HH:MM" |
| 默认拦截 Tab/Enter | 阻断后续操作 | 编辑模式才拦截，浏览模式放行 |
| 隐式状态切换 | 用户被困不知 | 状态可见 + 退出路径明确 |

## 七、抽象工程原则（最简版）

1. **副作用必须显式清理**——启动的进程 / 文件 / 监听器
2. **假设必须验证**——环境 / 依赖 / 字段 / 类型
3. **失败必须能回滚**——atomic commit + 清晰 message
4. **测试必须覆盖契约边界**——不只单元，要端到端
5. **错误必须追到根因**——不只修表象
6. **默认必须无副作用**——避免反 UX 陷阱
7. **经验必须沉淀**——同类 bug 出现 2 次就写守则

---

## 八、何时升级这份文档

- 同类 bug 出现 2 次以上 → 写新规则
- 修复花费 > 1 小时 → 写根因分析到 debugging-notes
- 发现新的第三方库陷阱 → 加到陷阱信号表
- 守则超过 15 条 → 合并 / 重构（保持精简）

---

## 不要做的事（Anti-Checklist）

- ❌ 不要修了 bug 不写复现测试
- ❌ 不要用 try-catch 吞错不报告
- ❌ 不要假设"应该能跑"而不验证
- ❌ 不要复制代码不验证逻辑
- ❌ 不要把"我假设"当成"事实"
- ❌ 不要在 commit message 写"修复 bug"而不说根因
- ❌ 不要让用户等你"再试一次"——立即查为什么失败
- ❌ 不要同一份经验分散到 5 个文档（精简到 1 份）
